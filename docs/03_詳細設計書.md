# 詳細設計書

## 1. モジュール詳細

---

### 1.1 main.c

#### 処理フロー

```
main(argc, argv)
  1. argc < 2 → usage() 表示してエラー終了
  2. cdbg_resolve_program(argv[1]) : カレントディレクトリ優先でパス解決
  3. spawn_argv[] に argv[1..] を格納（最大 256 要素）
  4. cdbg_init(&dbg)
  5. cdbg_set_run_target(&dbg, spawn_argv)
  6. cdbg_run(&dbg, NULL) : プログラムを起動してエントリで停止
  7. cdbg_repl(&dbg) : REPL ループ開始
  8. EXIT_SUCCESS で終了
```

---

### 1.2 process.c

#### cdbg_resolve_program

- 名前に `/` が含まれる → パスとして扱いそのままコピー
- 含まれない → `./<name>` が存在すればそちらを使用
- それ以外 → `PATH` 検索はせずそのまま（`execv` に委ねる）

#### cdbg_process_spawn

```
fork()
  子プロセス:
    1. setpgid(0, 0)       ← debuggee を独立プロセスグループへ
    2. ptrace(PT_TRACE_ME) ← 自身をトレース対象に登録
    3. malloc_stack_logging が true なら setenv("MallocStackLogging", "1")
    4. execv(argv[0], argv)

  親プロセス:
    *child_pid = pid; return 0
```

#### cdbg_process_wait

```c
do {
    ret = waitpid(pid, status, 0);
} while (ret == -1 && errno == EINTR);  // SIGINT による中断を再試行
```

### 1.3 memory.c

#### アクセス経路の優先順位

```
cdbg_mem_read(pid, addr, buf, len)
  1. task_for_pid() で Mach タスクポート取得
  2. 成功 → mach_vm_read() でまとめて読み取り
  3. 失敗 → ptrace(PT_READ_D) で 1 バイトずつ読み取り

cdbg_mem_write(pid, addr, buf, len)
  1. task_for_pid() で Mach タスクポート取得
  2. 成功 → mem_write_mach()
       2a. mach_vm_write() 直接試行
       2b. 失敗（保護あり）→ mach_vm_protect(R|W|COPY) → write → protect(R|X)
  3. 失敗 → ptrace(PT_WRITE_D) で 1 バイトずつ書き込み
```

**理由**: `mach_vm_write` は書き込み保護ページに対して失敗するため、`mach_vm_protect` で一時的に書き込み可能にする。これはブレークポイント（テキストセクション書き換え）に必要。

### 1.4 regs.c

#### レジスタ取得

```
cdbg_regs_get(pid, tid, regs)
  1. cdbg_threads_resolve_port(pid, tid, &thread)   // threads.c、tid==0ならtask_threads()[0]
  2. thread_get_state(thread, CDBG_THREAD_FLAVOR, &regs->native, &count)
  3. thread_get_state(thread, CDBG_NEON/FP_FLAVOR, &regs->neon/fp, &count)
     （同じ thread ポートを使い、mach_port_deallocate は両呼び出しの後に行う）
```

`regs.c` の全関数（`cdbg_regs_get`/`_set`/`_get_debug_state`/`_set_debug_state`/`_get_exception_state`/`_set_by_name`）は `pid_t pid` の次に `uint64_t tid` を引数に取る。旧実装の `primary_thread_for_task()`（プライマリスレッド固定）は廃止され、`cdbg_threads_resolve_port`（`threads.c`、詳細は 1.4b）に置き換えられた。

ARM64 専用実装（`arm_thread_state64_t` / `arm_neon_state64_t`）。

#### レジスタ一覧

| 種別 | レジスタ | Mach 構造体フィールド |
|---|---|---|
| 汎用 (GPR) | x0-x28, fp(x29), lr(x30), sp, pc, cpsr | `arm_thread_state64_t.__x[]`, `.__fp`, `.__lr`, etc. |
| NEON/FP | v0-v31 | `arm_neon_state64_t.__v[]` (`__uint128_t`) |
| FP 制御 | fpsr, fpcr | `arm_neon_state64_t.__fpsr`, `.__fpcr` |

#### 関数

| 関数 | 説明 |
|---|---|
| `cdbg_regs_get_by_name(regs, name, out)` | レジスタ名で値を取得（整数として返す） |
| `cdbg_regs_set_by_name(pid, regs, name, value, is_float, fvalue)` | レジスタ名で値を設定し thread_set_state で書き戻す |
| `cdbg_regs_get_debug_state` / `cdbg_regs_set_debug_state` | `ARM_DEBUG_STATE64`（`DBGWVR`/`DBGWCR` 等）の取得・設定。ウォッチポイント機構（watchpoint.c）が使用 |
| `cdbg_regs_get_exception_state` | `ARM_EXCEPTION_STATE64` から `FAR`（フォールトアドレス）・`ESR`（例外症候群レジスタ）を取得。ウォッチポイント発火要因の判定に使用 |

#### フレームアンワインド (cdbg_regs_frame_up)

```
1. FP != 0 かつ [FP+0]/[FP+8] が妥当（saved_FP > FP かつ戻り先アドレスが妥当）
     → PC = [FP+8], FP = [FP+0], SP = FP+16
2. それ以外 → LR（リンクレジスタ）が有効なら PC = LR とし、
     [旧FP]=saved_FP が読めれば FP を更新、SP = 旧SP+16
```

### 1.4b threads.c

スレッド列挙・スレッド識別・スレッド単位の実行凍結/再開を担当する。詳細設計は [02_基本設計書.md](02_基本設計書.md) の「7c. スレッド管理機構」を参照。

```c
typedef struct {
    uint64_t tid;    // THREAD_IDENTIFIER_INFO.thread_id（安定ID）
    uintptr_t pc;    // 一覧表示用のスナップショット
} cdbg_thread_entry_t;
```

| 関数 | 役割 |
|---|---|
| `cdbg_threads_list(pid, out, max, count_out)` | `task_threads()` の順序で全スレッドを列挙し、各スレッドの tid・PC を取得 |
| `cdbg_threads_resolve_port(pid, tid, port_out)` | tid を送信権に解決。`tid == 0` は `task_threads()[0]`（プライマリスレッド）を意味する。呼び出し側が `mach_port_deallocate` すること |
| `cdbg_threads_suspend_others(pid, keep_tid)` | `keep_tid` 以外の全スレッドを `thread_suspend` で凍結（`keep_tid == 0` はプライマリスレッドを維持) |
| `cdbg_threads_resume_others(pid, keep_tid)` | 上記で凍結したスレッドを `thread_resume` で再開 |

内部実装はいずれも `task_for_pid` → `task_threads` → 対象スレッド以外の送信権を `mach_port_deallocate` という共通パターンを使う（`enumerate_threads`/`deallocate_threads` static ヘルパー）。

### 1.5 breakpoint.c

ARM64 専用。トラップは `BRK #0`（`0xd4200000`、4バイト命令）を使う。

#### cdbg_bp_enable

```
1. cdbg_mem_read(pid, addr, &saved_inst, 4)
2. bp->addr = addr; bp->saved_inst = saved_inst; bp->enabled = true
3. cdbg_mem_write(pid, addr, &CDBG_TRAP_INST, 4)   // 0xd4200000
```

#### cdbg_bp_matches_pc

```c
return bp->enabled && bp->addr == pc;
```

BRK 実行後、Mach 例外から得られる PC は BRK 命令の次のアドレスを指すため、ヒット処理側で明示的に `bp->addr` へ戻す（下記）。

#### ブレークポイントヒット後の再開処理（debugger.c 内、handle_breakpoint_hit）

```
1. cdbg_bp_disable() : 元の命令を復元
2. cdbg_regs_set_pc(bp->addr) : PC を BRK 命令のアドレスに戻す
3. cdbg_regs_set() : PC をデバッギに書き戻す
4. （次の continue 時に）シングルステップで元の命令を実行 → cdbg_bp_enable() で再設置
```

### 1.5b watchpoint.c

ARM64 ハードウェアデバッグレジスタ（`DBGWVRn_EL1`/`DBGWCRn_EL1`、最大 `CDBG_MAX_WATCHPOINTS`=4 本）を用いる。詳細は [02_基本設計書.md](02_基本設計書.md) の「7b. ウォッチポイント機構」を参照。

```c
typedef struct cdbg_watchpoint {
    uintptr_t addr;
    size_t size;                 // 1/2/4/8 バイト、8バイト境界をまたがない
    cdbg_watch_kind_t kind;      // READ=1 / WRITE=2 / ACCESS=3(=READ|WRITE)
    bool enabled;
    int slot;                    // 0..CDBG_MAX_WATCHPOINTS-1（配列上の番号とは独立）
    char expr[128];
    uint64_t last_value;
    bool has_last_value;
} cdbg_watchpoint_t;
```

#### 主要関数

| 関数 | 役割 |
|---|---|
| `cdbg_watch_arm(pid, tid, slot, addr, size, kind)` | 指定スレッドの WVR/WCR を設定して有効化。読み戻して実装確認（未実装なら失敗） |
| `cdbg_watch_disarm(pid, tid, slot)` | 指定スレッドの WCR の enable ビットのみ落として無効化（他スロットは触らない） |
| `cdbg_watch_addr_hit(wp, far)` | FAR がウォッチポイントのアドレス範囲 `[addr, addr+size)` に入るか判定 |
| `cdbg_watch_is_watchpoint_esr(esr, &is_write)` | ESR の EC が `0x34`/`0x35`（ウォッチポイント例外）かを判定し、WnR ビットで読み/書きを返す |

`cdbg_watch_arm`/`cdbg_watch_disarm` は `tid` 引数を追加しただけで、他のロジックは単一スレッド時と同一。全スレッドへの腕装/解除は debugger.c 側の `arm_watchpoint_all_threads`/`disarm_watchpoint_all_threads`（`cdbg_threads_list` で列挙したスレッドをループして呼び出す）が担当する。

#### debugger.c 側の発火処理

```
try_report_watchpoint_hit(dbg):
  1. cdbg_regs_get_exception_state で FAR/ESR を取得
  2. cdbg_watch_is_watchpoint_esr で判定できなければ false を返す（他要因のトラップ）
  3. FAR が一致するウォッチポイントが見つかれば handle_watchpoint_hit、
     見つからなければ handle_unresolved_watchpoint_hit（幅の広い命令が
     複数ウォッチポイントの領域をまたいだ場合）
  4. どちらの場合も step_past_all_watchpoints で
     「全ウォッチポイント無効化 → 1命令シングルステップ → 全て再有効化」
     を行ってから戻る（無限再トラップの防止）
```

`report_stop`、`cdbg_step_next_line`、`cdbg_next_source_line` のいずれからも呼ばれ、`continue`/`step`/`next` のどの経路で発火しても同じ処理になる。

### 1.6 syms.c

#### シンボルロード戦略

```
cdbg_syms_load(executable_path)
  1. Mach-O をメモリマップ → __TEXT vmaddr を link_base として保存
  2. [executable_path, dSYM パス] それぞれに対して:
       syms_load_from_nm() : nm -n <path> の出力をパース
       成功したら return 0
  3. 全失敗 → Mach-O nlist_64 を直接パース (syms_load_macho)
  4. dSYM が存在すればそちらも試みる
```

#### シンボルフィルタリング条件 (sym_name_interesting)

- 名前が空でない
- type が `?` または `-` でない
- 先頭文字が `.` または `L` でない（ローカルラベル除外）
- 先頭文字が印字可能文字

#### シンボル名検索 (cdbg_syms_lookup_name)

```
入力名が '_' で始まらない場合、先頭に '_' を付けて検索
（C の external シンボルは Mach-O で先頭 _ が付く慣習）
```

#### ASLR スライド算出 (cdbg_syms_update_slide)

```
mach_vm_region_recurse で VM マップを先頭から走査:
  実行可能(EXECUTE|READ) な最初のリージョン先頭 = 実行時 __TEXT 先頭
  slide = 実行時先頭 - link_base
```

### 1.8 式パーサ (expr.y / expr.l / expr_eval.c)

式の字句解析・構文解析は flex(`expr.l`)/bison(`expr.y`) で生成する。旧実装（手書き再帰降下パーサ）はこれに置き換えられた。

#### 文法 (expr.y)

演算子優先順位は bison の `%left`/`%right` 宣言で C の規則どおりに与える（下ほど優先順位が高い）。

```
%left OROR
%left ANDAND
%left '|'
%left '^'
%left '&'
%left TOK_EQ TOK_NE
%left '<' '>' LE GE
%left SHL SHR
%left '+' '-'
%left '*' '/' '%'
%right UNARY_PREC        # 単項 + - ! ~ & *
```

後置演算（メンバアクセス・添字）は再帰的な文法規則で表現するため、`a.b.c` や `p->a.b` のような多段アクセス、`arr[i]`（`i` は任意の式）のような式添字が自然にサポートされる：

```
postfix_expr: primary_expr
    | postfix_expr '.' IDENT
    | postfix_expr ARROW IDENT
    | postfix_expr '[' expr ']'
    ;
```

構文解析の結果は `cdbg_expr_node_t`（`include/expr_ast.h`）の AST として構築される（`CDBG_EXPR_NUM` / `IDENT` / `UNARY` / `BINARY` / `MEMBER` / `ARROW` / `INDEX` / `ADDR` / `DEREF`）。

命名衝突を避けるため、bison/flex 生成コードはすべて `%name-prefix="expr_"` / `%option prefix="expr_"` を付与している（同一バイナリ内にコマンド用の別グラマー `command.y`/`command.l` も存在するため）。

#### 字句解析 (expr.l)

数値リテラル（10進・16進・浮動小数点数）、識別子、`$`を除く演算子全般、`. -> [ ] ( )` をトークン化する。浮動小数点数の判定は旧実装と同じ方式（`strtoull` で整数を試し、後続が `. e E` なら `strtod` に切り替える）。

#### 評価 (expr_eval.c)

AST を2通りに評価する:

- `cdbg_expr_eval()`（`cdbg_expr_result_t` を返す、右辺値評価） — `print` の一般式フォールバック、`set` の右辺、`set $reg = expr` の右辺で使用。
- `cdbg_expr_eval_lvalue()`（アドレス・サイズ・符号・型文字列を返す、左辺値評価） — `print`/`set` のターゲット解決、`watch`/`rwatch`/`awatch` のターゲット解決で使用。旧 `resolve_lvalue`/`split_access_path`/`parse_array_index_expr` を置き換えた。

両者とも再帰的に AST を辿るため、`MEMBER`/`ARROW`/`INDEX`/`DEREF` は何段でもネストできる。ポインタの型情報が必要な箇所（`->` の左辺、`*` の被演算子、ポインタ添字）は内部ヘルパー `eval_as_pointer()` に集約している。

DWARF 変数解決・型サイズ推定・構造体メンバ検索などの下請け関数（`resolve_variable_address`, `type_element_size`, `dwarf_lookup_struct`, `lookup_struct_member`, `complete_var_type` 等）は `debugger.c` に定義されたままだが、`include/expr_shared.h` で非 static 宣言され `expr_eval.c` と共有される（旧実装で `expr.c`/`debugger.c` に重複していたコピーは削除済み）。

### 1.9 debugger.c

#### REPL ループ

```c
for (;;) {
    fputs("cdbg> ", stdout); fflush(stdout);
    if (fgets(line, sizeof(line), stdin) == NULL) {
        if (ferror(stdin) && errno == EINTR) {
            clearerr(stdin); putchar('\n'); continue;  // Ctrl+C
        }
        break;  // EOF → 終了
    }
    cdbg_command_t cmd = {0};
    cdbg_command_parse(line, &cmd);   // command.y / command.l
    switch (cmd.kind) { ... }         // コマンドディスパッチ
    cdbg_command_free(&cmd);
}
```

行の構造化解析（コマンド種別の判定と引数の切り出し）は flex(`command.l`)/bison(`command.y`) で行う。旧実装（`strtok` チェーンと `if/else if` の連鎖）はこれに置き換えられた。結果は `cdbg_command_t`（`include/command_ast.h`）として返る：

```c
typedef struct {
    cdbg_cmd_kind_t kind;          // CDBG_CMD_PRINT など
    char print_fmt;                 // print 専用: '/' の後の1文字、なければ '\0'
    cdbg_watch_kind_t watch_kind;   // watch 専用
    char *arg1, *arg2;              // 所有権あり(strdup)。省略時は NULL
    char *unknown_text;             // kind == CDBG_CMD_UNKNOWN のとき
} cdbg_command_t;
```

字句解析側 (`command.l`) は最初の1語を見てコマンド種別を判別し、その種別に応じて flex の開始条件（start condition）を切り替えて残りの字句規則を変える:

- `RAWLINE`（`help`/`run`/`set`/`show`/`del`/`watch`/`rwatch`/`awatch`/`delwatch`/`dis`/`list`/`print`）: 行の残り全体を1トークンとして取り込む（旧 `strtok(NULL, "\n")` と同等）。
- `TOKENS`（`break`/`x`/`lines`/`lists`/`syms`）: 空白区切りで1〜2個の単語トークンを返す（旧 `strtok(NULL, " \t")` と同等）。
- `IGNORE_REST`（引数を取らないコマンド）: 行の残りを読み捨てる。

`print`/`p`/`p/fmt`/`print/fmt` の書式文字は字句解析の時点で切り出し、構文解析側の `cmd_print_from_repl(dbg, fmt_char, rest)` に渡す。`break`/`x` の「必須引数が省略された」場合は文法上 `arg1 == NULL` として受理し、使用方法メッセージの表示はディスパッチ側（`cdbg_repl`）が従来どおり担当する。

各 `cmd_*` ハンドラ本体（デバッガのセマンティクス）はほぼ変更されておらず、引数の受け取り方（生文字列 → 構造化された `cdbg_command_t` のフィールド）のみが変わっている。

| 入力 | 処理関数 |
|---|---|
| `help` / `h` | `cmd_help` |
| `run` | `cmd_run` |
| `continue` / `c` | `cdbg_continue` + `cdbg_wait` |
| `step` / `s` | `cdbg_step_next_line` |
| `si` | `cdbg_single_step` + `cdbg_wait` |
| `next` / `n` | `cdbg_next_source_line` |
| `up` | `cdbg_frame_up` |
| `kill` | `stop_debuggee`（REPL は継続） |
| `regs` / `r` | `cdbg_refresh_regs` + `cdbg_regs_print` |
| `print` / `p` / `p/fmt` | `cmd_print_from_repl` |
| `set` | `cmd_set` |
| `show` | `cmd_show` |
| `tb` | `cmd_backtrace` |
| `leaks` | `cmd_leaks` |
| `break` / `b` | `cmd_break` |
| `del` / `delete` | `cmd_del` |

#### ステップ制御

**step (ソース行ステップイン)**

```
cdbg_step_next_line:
  current_line = lineno_lookup(pc)
  ループ:
    cdbg_single_step → cdbg_wait
    new_line = lineno_lookup(new_pc)
    new_line != current_line → 停止
    ブレークポイントにヒット → 停止
```

**next (ソース行ステップオーバー)**

```
cdbg_next_source_line:
  target_line = lineno_lookup_next_line_after_pc(pc)
  target_addr に一時ブレークポイントを設置 → continue → 到達したら削除
  ブレークポイント未解決の場合は step と同様のシングルステップループ
```

#### leaks コマンド (cmd_leaks)

```
1. malloc_stack_logging が false → 警告を出力（スタックトレースなしで続行）
2. snprintf(cmd, "leaks %d 2>&1", pid)
3. popen(cmd, "r") で leaks 出力をそのまま stdout に転送
4. leaks が見つからない場合（"leaks not found" 出力）はエラーメッセージ
```

#### 停止時の PID/TID 表示とヒットスレッド判定

ブレークポイントヒット・ステップ完了など、プロセスが停止するたびに以下を表示する：

```
[PID: 12345  TID: 67890]
```

`report_stop(dbg)` は表示前に、実際にどのスレッドが停止要因だったかを判定する（1スレッドしかない場合はこの判定は実質no-op）：

```
report_stop(dbg):
  1. hit_tid = find_breakpoint_hit_thread(dbg)
       cdbg_threads_list で全スレッドを列挙 → 各スレッドのPCを
       有効なブレークポイントのアドレスと照合
  2. hit_tid == 0 なら hit_tid = find_watchpoint_hit_thread(dbg)
       全スレッドを列挙 → 各スレッドの cdbg_regs_get_exception_state で
       ESR を取得 → cdbg_watch_is_watchpoint_esr で判定
  3. hit_tid != 0 なら dbg->current_tid = hit_tid
  4. cdbg_threads_suspend_others(dbg->pid, dbg->current_tid)
       選択中スレッド以外を Mach thread_suspend で凍結
  5. cdbg_refresh_regs(dbg) : 選択中スレッドのレジスタを取得
```

- `resolve_display_tid(dbg)` : `dbg->current_tid` が設定されていればそれを、未設定（0）なら `cdbg_threads_list` の先頭スレッドの tid を返す（旧 `get_primary_thread_id` を置き換え）
- `print_stop_header(dbg)` : `resolve_display_tid` の結果を `printf("[PID: %d  TID: %llu]\n", ...)` で出力
- 呼び出し箇所: `report_stop`, `cdbg_step_next_line`, `cdbg_next_source_line`, 初回エントリ停止（`cdbg_run` 内）
- 複数スレッドが同時に同じアドレスで停止した場合は最初に見つかったものを採用する（既知の簡略化）

`cdbg_continue` は選択中スレッド以外の凍結を解除してから `PT_CONTINUE` する（`cdbg_threads_resume_others(dbg->pid, dbg->current_tid)`）。`cdbg_single_step`（`step`/`next`/`si` が使う `PT_STEP`）は凍結状態をそのまま維持して選択中スレッドのみを進める。

その後、ソースコードまたは逆アセンブルを `print_stop_location` で表示する（ソース情報がある場合はソース、ない場合は `dis`）。

#### set コマンド (cmd_set)

```
"var = expr" 形式をパース:
  1. 左辺が '$' で始まる → レジスタ代入モード
       a. cdbg_expr_eval で右辺を評価（is_float, fvalue を含む）
       b. cdbg_refresh_regs でレジスタ読み込み
       c. cdbg_regs_set_by_name(pid, regs, reg_name, value, is_float, fvalue) で書き込み
  2. それ以外 → 変数代入モード
       a. cdbg_expr_eval_lvalue（expr.y/expr.l/expr_eval.c）で左辺を解析し
          書き込みアドレス・サイズ・型を解決（変数名、.member、->member、[n] は
          任意の深さでネスト可能）
       b. cdbg_expr_eval で右辺を評価
       c. 型サイズに応じて cdbg_mem_write
```

#### help コマンド (cmd_help)

```
cdbg_help_entry_t {
    const char *category;  // "General" / "Execution" / "Inspection" / "Breakpoints" / "Settings"
    const char *names;     // "break, b"
    const char *usage;     // "break <addr|name|file:line|line>"
    const char *summary;   // 1行の概要
    const char *detail;    // NULL または複数行の詳細テキスト
}
```

| 関数 | 役割 |
|---|---|
| `print_help_entry` | `help` 一覧用：`  %-22s  %s\n` で names + summary を1行表示 |
| `print_help_entry_detail` | `help <cmd>` 用：usage + summary + detail（2スペースインデント）を表示 |
| `print_help_all` | カテゴリ見出し + 全コマンド1行一覧 + 式サンプル集 |
| `help_name_matches` | `names` フィールドのカンマ区切りトークンまたは `usage` 先頭との前方一致で検索 |

全コマンドに `detail` テキストを持つ。`help <cmd>` で書式例・サブコマンド・注意事項を確認できる。

#### kill コマンド

```
1. pid <= 0 or state == IDLE → "No process is running." を stderr に出力して return
2. killed_pid = dbg->pid を保存
3. stop_debuggee(dbg) : ptrace(PT_KILL) → cdbg_process_wait → pid=0, state=IDLE
4. printf("Process %d killed.\n", killed_pid)
5. REPL を継続（debugger は終了しない）
```

`quit` との違い: `kill` はデバッギを終了するが REPL に戻る。`quit` はデバッギを終了したうえでデバッガ自体も終了する。

#### stop_debuggee（quit / kill 時に呼ばれる）

```
1. pid <= 0 → return 0（state だけ IDLE にする）
2. state != IDLE の場合:
     a. ptrace(PT_DETACH, pid, 1, 0) : トレースを解除し通常の子プロセスに戻す
     b. kill(pid, SIGKILL)
     c. cdbg_process_wait() : zombie 回収
3. pid = 0; state = IDLE; breakpoints/watchpoints すべて disabled
```

**PT_DETACH を先に呼ぶ理由**: マルチスレッドの debuggee が ptrace 停止中の状態で、そのまま `kill(SIGKILL)`（または `ptrace(PT_KILL)`）だけを送ると、このカーネル上では `waitpid()` が無期限にハングすることが実測で確認されている（シグナルが ptrace の再開機構を経由してしまい、トラップ中の1スレッドにしか確実に届かないためと推測される）。`PT_DETACH`（"PT_CONTINUE と同様だがもはやトレースされない"）を先に呼んでプロセスを通常の非トレース状態に戻してから `kill(SIGKILL)`+`waitpid()` を行うことで、この問題を回避する。単一スレッドの単純なケースでも含めて20回・30回の連続実行で100%再現なく動作することを確認済み（`ptrace(PT_KILL)` 単体では実測で約10%成功率にとどまった）。

## 2. エラー処理方針

- すべての関数は `0` を成功、`-1` をエラーとして返す（`bool` 返しを除く）
- ptrace・Mach API のエラーは `perror` または `fprintf(stderr, ...)` で報告
- REPL コマンドのエラーは stderr に出力し、REPL を継続する
- `cdbg_continue` / `cdbg_wait` がエラーを返した場合は REPL を終了する（`return -1`）
