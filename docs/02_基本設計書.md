# 基本設計書

## 1. システム概要

`cdbg` は macOS 上で動作する対話型 C デバッガである。`ptrace` と Mach カーネル API を組み合わせて debuggee を制御し、DWARF デバッグ情報を解析することで変数参照・ソース表示・ブレークポイントを実現する。

## 4. 状態機械

debugger は debuggee の状態を `cdbg_state_t` で管理する。

```
      run/spawn
  ┌─────────────────────────────────────────┐
  │                                         ▼
IDLE ◀────────────── exited/killed ──── RUNNING
  │                                         │
  │                                    waitpid停止
  │                                         │
  └──────────────── kill ──────────── STOPPED
                                            │
                           continue/step ───┘
```

| 状態 | 値 | 意味 |
|---|---|---|
| `CDBG_STATE_IDLE` | 0 | 対象プロセスなし |
| `CDBG_STATE_RUNNING` | 1 | 対象プロセス実行中 |
| `CDBG_STATE_STOPPED` | 2 | ptrace 停止中（ブレークポイント・シグナル等） |

## 5. 主要データ構造

### 5.1 cdbg_t（デバッガ中心構造体）

```c
typedef struct cdbg {
    pid_t            pid;                         // debuggee の PID
    cdbg_state_t     state;                       // 実行状態
    int              wait_status;                 // waitpid ステータス
    uint64_t         current_tid;                 // 選択中スレッドの安定ID（0=未選択、プライマリスレッド扱い）
    cdbg_regs_t      regs;                        // レジスタスナップショット（選択中スレッドのもの）
    cdbg_breakpoint_t breakpoints[64];            // ブレークポイント配列
    size_t           breakpoint_count;
    cdbg_lineno_t    lineno;                      // DWARF 行情報
    cdbg_syms_t      syms;                        // シンボルテーブル
    char             executable_path[1024];
    char             debug_info_path[1024];       // dSYM パス
    char             run_argv_storage[64][1024];  // 実行引数ストレージ
    char            *run_argv[65];
    size_t           run_argc;
    bool             print_pretty;               // pretty print フラグ
    bool             malloc_stack_logging;        // MallocStackLogging フラグ
    cdbg_language_t  language;                   // 式評価言語
} cdbg_t;
```

## 6. 起動シーケンス

```
main()
  │
  ├─ cdbg_resolve_program()   : パス解決
  ├─ cdbg_init()              : 構造体初期化

  │    └─ entry まで実行して停止

```

## 7. ブレークポイント機構

### 設定

1. 対象アドレスの命令（4 バイト）を `cdbg_mem_read` で退避
2. `BRK #0`（`0xd4200000`）を書き込む
3. `cdbg_breakpoint_t` に {addr, saved_inst, enabled=true} を記録

### ヒット時

1. `waitpid` が停止を通知
2. BRK 実行後は PC が命令の直後を指すため、ブレークポイントのアドレスまで戻す
3. トラップ命令を元の命令に復元
4. ブレークポイントを一時無効化 → シングルステップで元の命令を実行 → 再有効化

---

## 7b. ウォッチポイント機構

`watchpoint.c` が ARM64 のハードウェアデバッグレジスタ（`DBGWVRn_EL1` / `DBGWCRn_EL1`、最大 4 本）を用いて実装する。ソフトウェア的なメモリ比較やシングルステップは使わない。

### 設定 (`cdbg_watch_arm`)

1. `thread_get_state(ARM_DEBUG_STATE64)` で現在のデバッグレジスタ状態を取得
2. 対象アドレスを 8 バイト境界に整列した値を `WVR` に設定
3. `WCR` に enable・特権レベル(EL0のみ)・読取/書込選択(LSC)・バイトマスク(BAS)を設定
4. `thread_set_state` で書き戻し、読み戻して実装されているレジスタ本数を検証（未実装なら失敗を返す）

### 発火時

ARM64 のウォッチポイント例外は対象命令の実行前に発生し、PC は命令実行後まで進まないため、そのまま `continue` すると同じ命令で無限に再トラップする。そこで発火のたびに:

1. `thread_get_state(ARM_EXCEPTION_STATE64)` で `FAR`（フォールトアドレス）と `ESR`（例外症候群レジスタ）を取得
2. `ESR` の EC フィールドがウォッチポイント例外（`0x34`/`0x35`）であることを確認し、`WnR` ビットで読み/書きを判定
3. `FAR` を各ウォッチポイントのアドレス範囲と照合してヒットしたものを特定
   （複数ウォッチポイントが隣接し、幅の広いロード/ストア命令が一括でまたぐ場合は特定できないことがあるため、その場合は「特定不能」として扱いつつ処理は継続する）
4. 全ウォッチポイントを一旦無効化 → シングルステップで対象命令を実際に実行 → 再度有効化

書き込みウォッチポイントの Old/New 値は、この手順の前後でメモリを読むことで取得する（前＝Old、シングルステップ後＝New）。

ARM64 のハードウェアデバッグレジスタはスレッドごとの状態であるため、`watch`/`rwatch`/`awatch` 設定時は、その時点で `threads.c` が列挙できる全スレッドに対して腕装する（`arm_watchpoint_all_threads`）。停止のたびに新規スレッドが検出された場合は、既存の有効なウォッチポイントをそのスレッドにも追加で腕装する。ただし前回停止からスレッド生成後最初の停止までの間は、その新規スレッドでのヒットを取りこぼす可能性がある（スレッド生成を直接フックする手段がないための近似）。

## 7c. スレッド管理機構

`threads.c` が Mach API（`task_threads`, `thread_info(THREAD_IDENTIFIER_INFO)`, `thread_suspend`/`thread_resume`）を用いて実装する。

### スレッド識別

スレッドは Mach ポート（`thread_act_t`）ではなく `THREAD_IDENTIFIER_INFO.thread_id`（プロセス内で安定した 64bit ID）で識別する。ポートは `task_threads()` を呼ぶたびに新しい送信権として得られ、長期保持を前提にできないためである。`cdbg_t.current_tid` にはこの安定 ID を保持する（`0` は「未選択＝プライマリスレッド」を意味する）。

### レジスタ操作のスレッド対応

`regs.c` の全関数（`cdbg_regs_get`/`_set`/`_get_debug_state`/`_set_debug_state`/`_get_exception_state`）は `pid_t pid` の次に `uint64_t tid` を引数に取る。内部で `cdbg_threads_resolve_port(pid, tid, &port)` を呼び、`tid == 0` の場合は `task_threads()[0]`（プライマリスレッド）にフォールバックする。

### ヒットスレッドの判定

`report_stop()`（debugger.c）は停止のたびに以下を行う。

```
report_stop()
  ├─ find_breakpoint_hit_thread()
  │    全スレッドを列挙 → 各スレッドの PC を有効なブレークポイントのアドレスと照合
  ├─ 一致がなければ find_watchpoint_hit_thread()
  │    全スレッドを列挙 → 各スレッドの ESR を取得 → ウォッチポイント例外かを判定
  ├─ ヒットスレッドが見つかれば current_tid を更新
  ├─ cdbg_threads_suspend_others(pid, current_tid) で選択中スレッド以外を凍結
  └─ cdbg_refresh_regs() で選択中スレッドのレジスタを取得
```

複数スレッドが同時に同じアドレスで停止した場合は最初に見つかったものを採用する（既知の簡略化）。`[PID: x TID: y]` 表示にはこの判定結果（またはユーザーが `thread` コマンドで選択したスレッド）の tid を用いる。

### スレッド単位のステップ実行

`report_stop()` で選択中スレッド以外を凍結した状態を保ったまま、REPL はユーザーコマンドを待つ。`continue` 実行時に `cdbg_threads_resume_others(pid, current_tid)` で凍結を解除してから `PT_CONTINUE` する。`step`/`next`/`si`（内部的に `PT_STEP` を使う）は選択中スレッドのみを対象に単一命令/単一行を進め、他スレッドは凍結されたままなので実行が進まない。

### REPL コマンド

- `show threads` — 全スレッドを Num（表示番号）/Cur（選択中マーク）/TID/PC で一覧表示する。
- `thread <n>` — 表示番号 `<n>` のスレッドを選択し、`current_tid` を更新して以後の `regs`/`print`/`tb` 等の対象を切り替える。

### 既知の制限

ハードウェアウォッチポイントが有効な状態で、監視対象とは別のスレッドが同時に活発に実行を続けている場合、`continue` 内部で行うウォッチポイント一時解除→シングルステップ→再腕装の手順（`step_past_all_watchpoints`）がまれに（実測で 20〜35% 程度）ハングすることが確認されている。`sample`(1) による調査では、ハングする箇所は実行ごとに異なり（`step_over_disabled_breakpoint` 内、`step_past_all_watchpoints` 内、通常の `continue`+`waitpid` サイクル内、いずれの場合もある）、macOS のカーネル/ptrace 実装がハードウェアデバッグレジスタの書き換えと他スレッドの ptrace ステップ/継続の組み合わせを正しく直列化できていないことに起因すると推測される。この問題を再現しない範囲（ウォッチポイント未使用、または単一スレッドでのウォッチポイント使用）は影響を受けない。デバッガ自身がこのハングから自律的に回復する手段はなく、外部から `kill -9` する必要がある。

## 8. メモリアクセス戦略

```
cdbg_mem_read / cdbg_mem_write
  │
  ├─ task_for_pid 成功 → mach_vm_read / mach_vm_write（高速）
  │    書き込み時: 保護 (R+W) を一時変更して書き込み後に (R+X) に戻す
  │
  └─ 失敗 → ptrace PT_READ_D / PT_WRITE_D（1 バイトずつ）
```

## 9. シンボル・ASLR 対応

```
cdbg_syms_load()
  ├─ Mach-O をメモリマップ → __TEXT セグメントのリンク時ベースアドレスを取得
  ├─ nm コマンドでシンボルテーブルを文字列解析（第一優先）
  └─ 失敗時は Mach-O nlist_64 を直接パース

cdbg_syms_update_slide()
  └─ mach_vm_region_recurse で実行可能領域先頭を探索 → slide = 実行時先頭 - link_base

runtime_addr = link_addr + slide
```

## 10. DWARF 行情報

```
cdbg_lineno_load()
  └─ dwarfdump --debug-line を実行して出力をパース
       → cdbg_line_entry_t[] {address, line, column, file}

cdbg_lineno_update_slide()
  └─ シンボルと同様に slide を設定

runtime_pc → (pc - slide) → lookup → file:line
```

## 11. シグナル処理

```
SIGINT 受信（Ctrl+C）
  │
  ├─ state == RUNNING → kill(pid, SIGINT) で debuggee に転送
  │     → ptrace がシグナルを捕捉 → waitpid 復帰 → STOPPED 状態
  │
  └─ state != RUNNING（プロンプト待機中）
        → fgets が EINTR で中断 → clearerr + 改行 + プロンプト再表示

子プロセス分離:
  cdbg_process_spawn で setpgid(0, 0) を呼び、
  debuggee を独立したプロセスグループに移す。
  端末の Ctrl+C が両者に届かないよう分離した上で、
  シグナルハンドラが明示的に転送する。
```
